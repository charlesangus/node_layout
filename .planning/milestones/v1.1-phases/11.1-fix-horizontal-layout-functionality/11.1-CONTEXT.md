# Phase 11.1: fix horizontal layout functionality - Context

**Gathered:** 2026-03-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix two bugs in the horizontal B-spine layout that cause incorrect geometry when the horizontal chain is embedded inside a larger vertical B-spine tree:

1. **Left-extent overlap**: the horizontal chain's leftmost nodes extend past (to the left of) the main vertical spine node (m1), overlapping it.
2. **Dot Y misalignment**: the output dot is placed below m2 (root) at a fixed gap, but it should sit at m1's Y level so the wire from m1 to the dot is horizontal.

Also close the 4 debug documents from Phase 11's UAT that are marked "verification: not yet applied" ‚ÄĒ those fixes are confirmed in the codebase.

The previously-diagnosed bugs (duplicate dot, no dot after Layout Selected Horizontal, downstream-node replay not firing) are already fixed in the code. Phase 11.1 does NOT re-implement those.

</domain>

<decisions>
## Implementation Decisions

### Bug 1 ‚ÄĒ Left-extent overlap (spine_x computation)

- **Both** `layout_upstream` and `layout_selected` trigger the overlap when the horizontal chain is upstream of a vertical spine node (m1).
- **Fix approach**: walk the `spine_set` to count spine nodes, compute `step_x = horizontal_subtree_gap * scheme_multiplier` (same formula used inside `place_subtree_horizontal`), and use the total leftward extent to push `spine_x` far enough right that the **leftmost spine node** lands right of the consumer with a clean gap.
- The formula result: `spine_x` must be large enough that `spine_x - (n - 1) * step_x >= consumer.xpos() + consumer.screenWidth() + gap` (where n = spine node count).
- This fix applies to the `root is not original_selected_root` branch in both `layout_upstream` (‚Čąline 1313) and `layout_selected` (‚Čąline 1474).
- When the selected node IS the horizontal root (no ancestor walk needed), `spine_x = root.xpos() / root.ypos()` remains unchanged.

### Bug 2 ‚ÄĒ Output dot Y alignment

- When a downstream consumer exists (m1), the output dot's **Y must be centred on the consumer's Y**:
  `dot_y = consumer_node.ypos() + (consumer_node.screenHeight() - dot.screenHeight()) // 2`
- The dot's **X stays centred on m2** (already correct):
  `dot_x = root.xpos() + (root.screenWidth() - dot.screenWidth()) // 2`
- The consumer knows its Y at call time because `layout_upstream` / `layout_selected` finish placing the vertical spine before calling `_place_output_dot_for_horizontal_root`.
- Apply in both paths inside `_find_or_create_output_dot` and `_place_output_dot_for_horizontal_root`:
  - New dot creation path (lines ‚Čą404‚Äď408)
  - Existing dot reposition path (lines ‚Čą462‚Äď464)
  - Reuse-check reposition path (lines ‚Čą387‚Äď390)
- When **no consumer exists** (standalone horizontal chain with no downstream node): keep the current "below root at loose gap" placement ‚ÄĒ no change.

<specifics>
## Specific Ideas

User-provided geometry diagram (target behaviour):

```
|  n  n  n
|  |  |  |
|  .--m--m2
|        |
m1-------.
|
|
```

- Left `|` bar = main vertical B-spine
- `m1` = node in the main vertical spine; its input(0) connects to the output dot
- `m2` = rightmost node of horizontal chain (root); its output connects DOWN to the output dot
- `m`, `n` = remaining horizontal spine nodes extending LEFT of m2
- Output dot (`_OUTPUT_DOT_KNOB_NAME`): centred horizontally on m2, centred vertically on m1
- Wire m2 ‚Üí dot is VERTICAL (same X); wire m1 ‚Üí dot is HORIZONTAL (same Y)
- The ENTIRE horizontal chain (n, m, m2) sits to the RIGHT of m1; no node overlaps the main spine

</specifics>
