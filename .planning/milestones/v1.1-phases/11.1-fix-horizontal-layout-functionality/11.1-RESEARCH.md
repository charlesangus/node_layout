# Phase 11.1: fix horizontal layout functionality - Research

**Researched:** 2026-03-16
**Domain:** Python geometry bug fixes in node_layout.py (Nuke DAG layout engine)
**Confidence:** HIGH

## Summary

Phase 11.1 fixes two geometry bugs that manifest when the horizontal B-spine chain is embedded inside a larger vertical B-spine tree, plus closes four debug documents that were left in "not yet applied" status after Phase 11 completed.

Bug 1 is a left-extent overlap: the horizontal chain's leftmost nodes can land to the left of the main vertical spine node (m1) because `spine_x` is computed as `consumer.xpos() + consumer.screenWidth() + horizontal_gap`, which positions only the ROOT (m2, the rightmost node) just right of m1. Each additional spine node steps further left by `step_x`, so a chain of n nodes extends `(n-1) * step_x` pixels further left — potentially crossing m1's tile.

Bug 2 is a dot Y misalignment: all three dot-positioning code paths in `_find_or_create_output_dot` and `_place_output_dot_for_horizontal_root` currently place the dot at `root.ypos() + root.screenHeight() + dot_gap`, which is below m2. The dot should sit at m1's Y level so that the wire from m1 to the dot is horizontal (same Y), matching the target geometry diagram.

Both bugs are pure arithmetic changes to existing code with no new functions, no new imports, and no structural changes. The test framework is Python `unittest` with a nuke stub, run via `python3 -m unittest`.

**Primary recommendation:** Fix Bug 1 by pre-computing the left extent of the spine before committing to `spine_x`, and fix Bug 2 by threading `consumer_node` into the dot Y formula in all three placement paths.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Bug 1 — Left-extent overlap (spine_x computation)**

- Both `layout_upstream` and `layout_selected` trigger the overlap when the horizontal chain is upstream of a vertical spine node (m1).
- Fix approach: walk the `spine_set` to count spine nodes, compute `step_x = horizontal_subtree_gap * scheme_multiplier` (same formula used inside `place_subtree_horizontal`), and use the total leftward extent to push `spine_x` far enough right that the leftmost spine node lands right of the consumer with a clean gap.
- The formula result: `spine_x` must be large enough that `spine_x - (n - 1) * step_x >= consumer.xpos() + consumer.screenWidth() + gap` (where n = spine node count).
- This fix applies to the `root is not original_selected_root` branch in both `layout_upstream` (line ~1313) and `layout_selected` (line ~1474).
- When the selected node IS the horizontal root (no ancestor walk needed), `spine_x = root.xpos() / root.ypos()` remains unchanged.

**Bug 2 — Output dot Y alignment**

- When a downstream consumer exists (m1), the output dot's Y must be centred on the consumer's Y:
  `dot_y = consumer_node.ypos() + (consumer_node.screenHeight() - dot.screenHeight()) // 2`
- The dot's X stays centred on m2 (already correct):
  `dot_x = root.xpos() + (root.screenWidth() - dot.screenWidth()) // 2`
- The consumer knows its Y at call time because `layout_upstream` / `layout_selected` finish placing the vertical spine before calling `_place_output_dot_for_horizontal_root`.
- Apply in both paths inside `_find_or_create_output_dot` and `_place_output_dot_for_horizontal_root`:
  - New dot creation path (lines ~404–408)
  - Existing dot reposition path (lines ~462–464)
  - Reuse-check reposition path (lines ~387–390)
- When no consumer exists (standalone horizontal chain with no downstream node): keep the current "below root at loose gap" placement — no change.

### Pattern 1: spine_x Pre-Computation (Bug 1)

**What:** Before calling `place_subtree_horizontal`, compute how many spine nodes exist, then add `(n - 1) * step_x` to push `spine_x` far enough right that the leftmost node still clears the consumer.

**When to use:** Only in the `root is not original_selected_root` branch (both `layout_upstream` and `layout_selected`). The standalone case (`else` branch) does not have a consumer to overlap with.

**Formula:**
```python
# Inside the `root is not original_selected_root` branch:
step_x = int(current_prefs.get("horizontal_subtree_gap") * root_scheme_multiplier)
# Count spine nodes (replay_spine_set is already built above this point)
spine_node_count = len(replay_spine_set)  # or root_spine_set in layout_selected
leftward_extent = (spine_node_count - 1) * step_x
horizontal_gap = step_x  # minimum gap between leftmost node and consumer right edge
consumer = original_selected_root
spine_x = consumer.xpos() + consumer.screenWidth() + horizontal_gap + leftward_extent
spine_y = consumer.ypos()
```

**Note on `left_extents`:** `place_subtree_horizontal` computes `left_extents[i]` to handle side subtrees that extend past a spine node's left edge. The CONTEXT.md explicitly leaves this as Claude's discretion — it is safe to treat it as an out-of-scope edge case for this phase (the fix addresses the primary spine-node-overlap bug; side-subtree overlap is a secondary edge case).

### Pattern 2: Consumer-Aligned dot_y (Bug 2)

**What:** When a consumer is known, substitute the "below root" Y with the consumer's Y, centering the dot tile on the consumer tile.

**When to use:** In all three dot-position code paths, but only when a consumer is available. The no-consumer path (standalone horizontal chain) keeps the existing loose-gap formula.

**Formula:**
```python
# dot_x: unchanged — centred on root (m2)
dot_x = root.xpos() + (root.screenWidth() - dot.screenWidth()) // 2
# dot_y: centred on consumer (m1)
dot_y = consumer_node.ypos() + (consumer_node.screenHeight() - dot.screenHeight()) // 2
```

**Threading consumer into `_find_or_create_output_dot`:** This function already receives `consumer_node` as a parameter. The two fix sites inside it (reuse-check reposition at ~387–390, new dot creation at ~404–408) can use `consumer_node` directly without any signature change.

**Threading consumer into `_place_output_dot_for_horizontal_root`:** This function finds `consumer_node` via the `all_nodes` scan (lines ~446–456). In the `existing_dot is not None` reposition branch (~458–465), `consumer_node` may still be `None` if only the dot was found. The fix requires the scan to find both `existing_dot` AND `consumer_node` before the early return — the scan loop must not short-circuit after finding `existing_dot`.

### Anti-Patterns to Avoid

- **Do not use `is` for Nuke node identity.** All identity checks must use `id(a) == id(b)`. This was the root cause of the duplicate dot bug (fixed in 6f4f050) and must not be reintroduced.
- **Do not read prefs inside recursive functions.** `current_prefs` is passed as a parameter from the entry point; do not call `node_layout_prefs.prefs_singleton.get(...)` inside `place_subtree_horizontal` for the new formula — use the `current_prefs` already in scope at the call site.
- **Do not change the no-consumer path.** When `consumer_node is None`, the existing "below root at loose gap" dot_y formula is correct and must not be changed.
- **Do not mutate `replay_spine_set` / `root_spine_set` size before counting.** These sets are built by the while-cursor loop immediately before the `spine_x` block; `len()` of the already-built set gives the spine node count.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Spine node count | Walking spine a second time | `len(replay_spine_set)` / `len(root_spine_set)` | Sets are already built one block above the fix site |
| step_x formula | New spacing constant | `int(current_prefs.get("horizontal_subtree_gap") * root_scheme_multiplier)` | Same formula as line 568 inside `place_subtree_horizontal`; must stay in sync |
| Dot tile height for centering | Magic constant (12) | `dot.screenHeight()` | Dynamic; tile size can differ in different Nuke prefs |
| Consumer height for centering | Magic constant | `consumer_node.screenHeight()` | Dynamic |

## Common Pitfalls

### Pitfall 1: scan loop early return breaks consumer discovery

**What goes wrong:** The `existing_dot is not None` early return in `_place_output_dot_for_horizontal_root` (line 458) fires before `consumer_node` is found, leaving `consumer_node = None` when the dot_y fix needs it.

**Why it happens:** The scan visits nodes in `nuke.allNodes()` order. If the existing dot is found early, the loop continues (no break), but once `existing_dot is not None`, the `elif consumer_node is None:` branch still runs for remaining nodes. Actually the scan loop never breaks early — it always runs to completion. The early return is on lines 458–465, *after* the loop. So `consumer_node` is populated correctly before the early return.

**How to avoid:** Verify that the for-loop (lines 446–456) always runs to completion before any `return`. It does. The `consumer_node` will be populated correctly as long as the loop is not prematurely exited. No change to scan loop structure is needed.

**Warning signs:** If tests show `dot_y` = "below root" even when a consumer exists, check that the consumer was actually found (`consumer_node is not None`) before the `existing_dot is not None` return branch.

### Pitfall 2: step_x multiplied by snap_threshold

**What goes wrong:** Debug doc `horiz04` noted the `horizontal_subtree_gap` formula might need `* snap_threshold`. The confirmed formula in `place_subtree_horizontal` at line 568 is `int(current_prefs.get("horizontal_subtree_gap") * scheme_multiplier)` — no snap_threshold. The same formula in the current `layout_upstream` / `layout_selected` fix blocks (post-commit 6658d7b) also omits snap_threshold.

**How to avoid:** Use `int(current_prefs.get("horizontal_subtree_gap") * root_scheme_multiplier)` — identical to line 568. Do not add `* snap_threshold`.

### Pitfall 3: spine_x not accounting for step_x including node widths

**What goes wrong:** `step_x` in `place_subtree_horizontal` is the gap between nodes' bounding boxes. The full advance per step inside `place_subtree_horizontal` includes both `step_x` and `prev_node.screenWidth()`. The left extent from root to the leftmost node's left edge is `sum of (step_x + spine_node[i].screenWidth())` for i in 1..n-1, not just `(n-1) * step_x`.

**Why it matters:** If all spine nodes have the same width `w`, the leftward extent from m2's left edge to the leftmost node's left edge is `(n-1) * (step_x + w)`. The current locked decision formula uses `(n-1) * step_x` — this only guarantees the *gap between bounding boxes* is at least `gap`, not that the leftmost node itself clears the consumer.

**How to avoid:** The CONTEXT.md formula `spine_x - (n-1) * step_x >= consumer.xpos() + consumer.screenWidth() + gap` is the stated requirement. Verify that `step_x` in this context means "gap + node width" or just "gap". Looking at `place_subtree_horizontal` lines 698–702: `cur_x = spine_x` for the root; the next spine node to the left is advanced by `step_x + spine_nodes[index+1].screenWidth()`. So the distance from root's left edge to the leftmost node's left edge is `sum((step_x + node.screenWidth()) for each intermediate step)`.

**Recommendation (Claude's discretion):** Use the conservative approach — compute the full leftward extent by walking `replay_spine_set` and summing `step_x + node.screenWidth()` for each spine node except the root. This is the correct safe formula. If CONTEXT.md's simpler `(n-1) * step_x` is explicitly chosen, document it as a simplification that only works when all spine nodes are the same width as the step_x gap.

### Pitfall 4: `_find_or_create_output_dot` reuse-check path missing consumer

**What goes wrong:** The reuse-check reposition path (lines ~387–390) already has `consumer_node` in scope but currently uses the `dot_gap` formula for `dot_y`. The fix is a one-line change to `dot_y`. The key insight: `consumer_node` is already in scope as a parameter — no need to modify the function signature.

**How to avoid:** Confirm the function signature: `_find_or_create_output_dot(root, consumer_node, consumer_slot, current_group, ...)`. `consumer_node` is the second parameter. All three fix sites within this function can use it directly.
