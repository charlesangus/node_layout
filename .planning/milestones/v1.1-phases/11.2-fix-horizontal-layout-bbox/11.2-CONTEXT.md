# Phase 11.2 Context

## What the user sees

Running **Layout Upstream** on a consumer node (m1) whose A-input chain has been laid
out horizontally:

1. The horizontal chain's **bbox overlaps m1** — there is no `horizontal_subtree_gap`
   between m1's right edge and the chain's leftmost extent.
2. **Other subtrees** (m1's B-input vertical chain) are placed **inside the horizontal
   chain's bbox**, because the layout engine doesn't respect the space the horizontal
   chain already occupies.

Subtrees always go to the **right** of the consumer in horizontal layout.

---

## Bug A — spine_x clearance mismatch

### Where: `layout_upstream` (and `layout_selected` replay branch)

The caller computes `spine_x` with:

```python
leftward_extent = sum(
    step_x + spine_nodes_ordered[i].screenWidth()
    for i in range(1, len(spine_nodes_ordered))
)
spine_x = consumer.xpos() + consumer.screenWidth() + step_x + leftward_extent
```

But inside `place_subtree_horizontal`, the actual advance formula (line 854) is:

```python
cur_x = cur_x - step_x - effective_widths[index + 1] - left_extents[index]
```

`effective_widths[i]` = max(spine_node.screenWidth(), right_extent_of_side_subtrees).
`left_extents[i]` = how far the current spine node's side subtrees extend LEFTWARD past
its own left edge.

**Mismatch**: when a spine node has a side subtree wider than itself, `effective_widths[i]
> screenWidth()`. The actual leftmost node lands further left than spine_x predicted —
eating into or eliminating the `step_x` clearance from m1.

### Fix A

After `place_subtree_horizontal` returns, compute the **actual left edge** of the placed
chain (bbox min_x over all placed nodes including side subtrees). If that left edge is
closer to m1's right edge than `horizontal_subtree_gap`, translate the entire chain
rightward by the deficit.

This "place then measure then shift" approach is unconditional and requires no changes
to `place_subtree_horizontal`'s internal advance logic.

Steps:
1. Call `place_subtree_horizontal(...)` at the provisional `spine_x`.
2. Collect all placed horizontal chain nodes (spine + side subtrees) via
   `collect_subtree_nodes(root)` plus any output dot.
3. Compute `actual_left = min(n.xpos() for n in chain_nodes)`.
4. `required_left = consumer.xpos() + consumer.screenWidth() + horizontal_subtree_gap`.
5. If `actual_left < required_left`, shift every chain node rightward by
   `required_left - actual_left`.

## Bug B — Phase 2 vertical subtrees land inside horizontal chain bbox

### Where: Phase 2 block in `layout_upstream` (~line 1423)

After Phase 1 places the horizontal chain, Phase 2 calls:

```python
place_subtree(
    original_selected_root,          # m1
    original_selected_root.xpos(), original_selected_root.ypos(),
    memo_phase2, ...,
    node_filter=vertical_filter,     # excludes horizontal chain nodes
)
```

`place_subtree` places m1's vertical inputs (B chain) centered above m1. The horizontal
chain's side subtrees are ALSO above m1 (placed above their spine node by
`place_subtree_horizontal`). Phase 2 has no knowledge of this occupied region.

Result: B-input nodes land inside the horizontal chain's side-subtree bbox.

### Fix B

After Fix A (horizontal chain is correctly positioned), compute the full horizontal
chain bbox. Pass the horizontal chain's right boundary as a `left_boundary` constraint
to Phase 2 — specifically: shift the vertical subtree's X anchor so that its rightmost
extent doesn't exceed `chain_bbox.min_x - horizontal_subtree_gap`.

Concrete approach:
1. After the chain is correctly placed (post Fix A shift), compute
   `chain_bbox = compute_node_bounding_box(chain_nodes)`.
2. `chain_left = chain_bbox[0]`  (min_x of the full chain including side subtrees).
3. Compute Phase 2 vertical subtree width:
   `phase2_w, _ = compute_dims(original_selected_root, memo_phase2, ..., node_filter=vertical_filter)`.
4. The rightmost point of the Phase 2 subtree when centered on m1 is approximately
   `m1.xpos() + (m1.screenWidth() + phase2_w) // 2`.
5. If that would exceed `chain_left - horizontal_subtree_gap`, shift the Phase 2 anchor
   leftward so the subtree's right edge is at `chain_left - horizontal_subtree_gap`.
6. Pass the adjusted X as the x-anchor to `place_subtree(...)`.

## Acceptance criteria

- Layout Upstream on m1 (A-input horizontal chain, B-input vertical chain):
  - `min_x(chain_nodes) >= m1.xpos() + m1.screenWidth() + horizontal_subtree_gap` (Bug A)
  - `max_x(b_chain_nodes) < min_x(chain_nodes) - horizontal_subtree_gap` (Bug B)
  - No existing tests regress (274 passing, 4 pre-existing Undo failures)

## Files to change

- `node_layout.py` — `layout_upstream` (primary), possibly `layout_selected` if its
  replay branch has the same spine_x mismatch (audit during implementation)

## Key constants / prefs

- `horizontal_subtree_gap` (default 250px) — gap between consumer right edge and
  horizontal chain left edge (Bug A) AND between horizontal chain left edge and B-chain
  right edge (Bug B)
- `horizontal_side_vertical_gap` (default 150px) — vertical gap between spine and
  first node of each side subtree (existing, not changed)

## Out of scope

- First-run horizontal layout (Layout Selected Horizontal) — that path is separate
- side-subtree left extents overlapping the consumer vertically — only X-axis overlap
  is in scope for this phase
