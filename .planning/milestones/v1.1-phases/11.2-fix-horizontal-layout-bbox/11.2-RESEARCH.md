## User Constraints (from CONTEXT.md)

### Locked Decisions

**Bug A fix — "place then measure then shift":**
After `place_subtree_horizontal` returns, compute the actual left edge of the placed chain
(bbox min_x over all placed nodes including side subtrees). If that left edge is closer to
m1's right edge than `horizontal_subtree_gap`, translate the entire chain rightward by the
deficit. Unconditional, no changes to `place_subtree_horizontal` internals.

Steps:
1. Call `place_subtree_horizontal(...)` at the provisional `spine_x`.
2. Collect all placed horizontal chain nodes (spine + side subtrees) via `collect_subtree_nodes(root)` plus any output dot.
3. Compute `actual_left = min(n.xpos() for n in chain_nodes)`.
4. `required_left = consumer.xpos() + consumer.screenWidth() + horizontal_subtree_gap`.
5. If `actual_left < required_left`, shift every chain node rightward by `required_left - actual_left`.

**Bug B fix — shift Phase 2 anchor leftward:**
After Fix A (chain correctly positioned), compute the full horizontal chain bbox. Pass
the chain's left boundary as a constraint to Phase 2 — shift the vertical subtree's X
anchor so its rightmost extent doesn't exceed `chain_bbox.min_x - horizontal_subtree_gap`.

Steps:
1. After the chain is correctly placed (post Fix A shift), compute `chain_bbox = compute_node_bounding_box(chain_nodes)`.
2. `chain_left = chain_bbox[0]` (min_x of the full chain including side subtrees).
3. Compute Phase 2 vertical subtree width: `phase2_w, _ = compute_dims(original_selected_root, memo_phase2, ..., node_filter=vertical_filter)`.
4. Rightmost point of Phase 2 subtree when centered on m1: `m1.xpos() + (m1.screenWidth() + phase2_w) // 2`.
5. If that exceeds `chain_left - horizontal_subtree_gap`, shift the Phase 2 anchor leftward.
6. Pass the adjusted X as the x-anchor to `place_subtree(...)`.

## Summary

Phase 11.2 fixes two geometric bugs in the horizontal layout replay path inside
`layout_upstream` (and the structurally identical block in `layout_selected`). Both bugs
stem from the same root cause: the caller computes `spine_x` using `node.screenWidth()`
but `place_subtree_horizontal`'s advance formula uses `effective_widths[i]`, which can be
larger than `screenWidth()` when side subtrees are wider than their spine node. This means
the placed chain's actual leftmost extent can be further left than predicted, eating into
or eliminating the `horizontal_subtree_gap` clearance.

Bug A is the direct clearance failure: the horizontal chain can overlap the consumer's
right edge. Bug B is a downstream consequence: Phase 2's vertical subtree is placed
centered above the consumer without knowing the horizontal chain already occupies space to
the right — the B-chain's rightmost extent can land inside the horizontal chain's bbox.

Both bugs are fully self-contained to `layout_upstream` (and the equivalent block in
`layout_selected`). No changes to `place_subtree_horizontal`, `compute_dims`, or
`place_subtree` are needed. The pattern is: place, then measure, then correct.

**Primary recommendation:** Implement the "place-then-measure-then-shift" pattern for Fix
A, then use the shifted chain's bbox as a right-boundary constraint for the Phase 2 anchor
in Fix B. Apply the same fix pattern to the structurally identical block in
`layout_selected`.

## Architecture Patterns

### Where both fixes live

Both bugs occur in the same two code blocks:

**`layout_upstream` (line ~1340–1457):** The primary entry point. Has two sub-branches:
- `if root is not original_selected_root:` — BFS found a horizontal ancestor upstream of
  the selected consumer (consumer = `original_selected_root`). Consumer is known here as
  the `consumer` local variable.
- `else:` — root IS the originally selected node. Consumer is found via the downstream
  scan as `downstream_consumer`.

**`layout_selected` (line ~1600–1713):** Structurally identical. The two sub-branches are:
- `if root is not original_selected_root:` — consumer is `original_selected_root`.
- `else:` — consumer is `downstream_consumer`.

In both functions, Phase 2 runs only in the `root is not original_selected_root` branch
(line 1423 in `layout_upstream`, line 1683 in `layout_selected`). Bug B only affects
this branch.

### Fix A: place-then-measure-then-shift pattern

```python
# After place_subtree_horizontal(...) returns:
chain_nodes = collect_subtree_nodes(root)
# Include the output dot in chain_nodes for bbox measurement.
# (Output dot created by _place_output_dot_for_horizontal_root, called next.)
# However, Fix A shift must happen BEFORE the output dot is placed — or include it
# after placement. Simplest: shift before dot placement, the dot is placed correctly
# by _place_output_dot_for_horizontal_root which reads root's updated position.

actual_left = min(n.xpos() for n in chain_nodes)
required_left = consumer.xpos() + consumer.screenWidth() + horizontal_subtree_gap
deficit = required_left - actual_left
if deficit > 0:
    for chain_node in chain_nodes:
        chain_node.setXpos(chain_node.xpos() + deficit)
```

The output dot is placed by `_place_output_dot_for_horizontal_root` which is called
AFTER `place_subtree_horizontal`. Since the dot's position derives from the root node's
final X/Y, it automatically lands in the correct place after the shift — no extra work
needed.

### Fix B: Phase 2 X-anchor clamping pattern

```python
# After Fix A shift, chain_nodes positions are final.
# Build chain_bbox now (output dot is placed by this point).
chain_all_nodes = collect_subtree_nodes(root)
# Optionally include output dot:
for _slot in range(original_selected_root.inputs()):
    _inp = original_selected_root.input(_slot)
    if _inp is not None and _inp.knob(_OUTPUT_DOT_KNOB_NAME) is not None:
        chain_all_nodes.append(_inp)
chain_bbox = compute_node_bounding_box(chain_all_nodes)
chain_left = chain_bbox[0]

# Phase 2: compute dims first to get phase2_w.
memo_phase2 = {}
phase2_w, _ = compute_dims(
    original_selected_root, memo_phase2, snap_threshold, node_count,
    node_filter=vertical_filter,
    scheme_multiplier=consumer_scheme_multiplier,
    per_node_h_scale=per_node_h_scale,
    per_node_v_scale=per_node_v_scale,
)

# Default Phase 2 anchor (centered on consumer, same as before).
phase2_anchor_x = original_selected_root.xpos()

# Clamp so rightmost extent of Phase 2 subtree doesn't enter chain bbox.
# When place_subtree places with x = phase2_anchor_x, the subtree width is
# phase2_w and it is centered starting at phase2_anchor_x. The rightmost point
# (when n==1 input, centered above consumer) is approximately:
#   phase2_anchor_x + phase2_w
# A safe bound: ensure phase2_anchor_x + phase2_w <= chain_left - horizontal_subtree_gap
max_right_for_phase2 = chain_left - horizontal_subtree_gap
if phase2_anchor_x + phase2_w > max_right_for_phase2:
    phase2_anchor_x = max_right_for_phase2 - phase2_w

place_subtree(
    original_selected_root,
    phase2_anchor_x, original_selected_root.ypos(),
    memo_phase2, snap_threshold, node_count,
    node_filter=vertical_filter,
    ...
)
```

**Important nuance on phase2_w vs actual right extent:** `compute_dims` returns the total
subtree width. For `place_subtree`, when there is exactly 1 input (n==1), the input is
placed directly above the root at the same X — the subtree occupies `[phase2_anchor_x,
phase2_anchor_x + phase2_w]`. For n >= 2 the rightward spread can be larger. The CONTEXT
formula `m1.xpos() + (m1.screenWidth() + phase2_w) // 2` is the centered-layout
rightmost point and should be used as specified.

### Anti-patterns to avoid

- **Modifying `place_subtree_horizontal` internals:** The fix is explicitly post-hoc.
  Do not adjust the advance formula or add lookahead inside the function.
- **Using `spine_nodes_ordered` to measure the chain:** `spine_nodes_ordered` contains
  only the spine, not side subtrees. The bbox must come from `collect_subtree_nodes(root)`
  which traverses all inputs recursively.
- **Computing chain_bbox before Fix A shift:** The bbox used for Bug B must reflect the
  post-Fix-A positions, not the pre-shift positions.
- **Omitting `layout_selected` fix:** The `layout_selected` replay block is structurally
  identical and has the same bugs. Both functions must be fixed in the same phase.

## Common Pitfalls

### Pitfall 1: Measuring chain before output dot is placed

**What goes wrong:** Fix A collects `collect_subtree_nodes(root)`, which does NOT include
the output dot (it is created by `_place_output_dot_for_horizontal_root`, called next).
If Fix A shift happens before the dot is placed, the dot is not in `chain_nodes` and
will not be shifted — but this is acceptable because the dot's position is computed fresh
by `_place_output_dot_for_horizontal_root` from root's post-shift position.

**Fix B** needs the output dot included in `chain_all_nodes` for the bbox, because the
dot sits below the root and may extend the min_x of the chain (if the dot is narrower
than the root, it doesn't change min_x; if the dot is centered and narrower, still safe).
By the time Fix B runs, the output dot has already been placed by
`_place_output_dot_for_horizontal_root`.

**How to avoid:** Place Fix A shift before `_place_output_dot_for_horizontal_root`. Collect
output dot into `chain_all_nodes` for Fix B AFTER the dot has been placed.

### Pitfall 2: Phase 2 node_filter must exclude output dot

**What goes wrong:** The existing code already adds the output dot to
`horizontal_block_node_ids` to prevent `place_subtree` from recursing through it. This
existing exclusion must be preserved — the Fix B anchor adjustment does not change this.

**How to avoid:** Do not remove the `_OUTPUT_DOT_KNOB_NAME` exclusion from
`horizontal_block_node_ids` in the Phase 2 block.

### Pitfall 3: Incorrect Phase 2 width formula

**What goes wrong:** `compute_dims` returns the full subtree width. When n >= 2 inputs,
`place_subtree` fans inputs rightward, so the actual rightmost extent is larger than
`phase2_anchor_x + phase2_w`. Using `phase2_anchor_x + phase2_w` underestimates the
rightmost spread in the multi-input case.

**CONTEXT.md specifies:** Use `m1.xpos() + (m1.screenWidth() + phase2_w) // 2` as the
rightmost point estimate. This is the centered-layout formula that matches how
`place_subtree` centers the subtree above the consumer when n==1. For the immediate
acceptance criteria (B-input vertical chain = single node), this is exact.

**How to avoid:** Follow the CONTEXT.md formula precisely. Mark the multi-input fan case
as a known approximation (acceptable per CONTEXT.md scope).

### Pitfall 4: `horizontal_subtree_gap` must be read from prefs (not a raw constant)

**What goes wrong:** `horizontal_subtree_gap` must be read as
`current_prefs.get("horizontal_subtree_gap")`, not hardcoded. The existing code already
does this; Fix A/B must follow the same pattern.

**How to avoid:** Reuse the already-computed `step_x` where possible (it equals
`int(current_prefs.get("horizontal_subtree_gap") * root_scheme_multiplier)`). For the
raw gap without scheme multiplier, read `current_prefs.get("horizontal_subtree_gap")`
directly.

### Pitfall 5: Fix not applied in the `else` branch (root IS the selected node)

**What goes wrong:** The `layout_upstream` function has two spine_x computation paths.
The `if root is not original_selected_root:` branch has consumer = `original_selected_root`.
The `else:` branch has consumer = `downstream_consumer`. Fix A must be applied in BOTH
branches when a consumer is identified. Phase 2 (Bug B) only runs in the if-branch, so
Fix B is if-branch only — but Fix A must cover both.

**How to avoid:** After each `place_subtree_horizontal` call that has a known consumer,
apply the clearance check. The else-branch ends with a `place_subtree_horizontal` call
only when `downstream_consumer is not None` — that is the correct guard.

```python
def compute_node_bounding_box(nodes):
    if not nodes:
        return None
    min_x = min(n.xpos() for n in nodes)
    min_y = min(n.ypos() for n in nodes)
    max_x = max(n.xpos() + n.screenWidth() for n in nodes)
    max_y = max(n.ypos() + n.screenHeight() for n in nodes)
    return (min_x, min_y, max_x, max_y)
```

## Open Questions

1. **Does the `else` branch (root IS selected) also need Fix B?**
   - What we know: Phase 2 only runs in the `if root is not original_selected_root` branch,
     which always has an identified consumer. The `else` branch has no Phase 2 block.
   - What's unclear: Whether `layout_selected` or future code adds Phase 2 to the else-branch.
   - Recommendation: Fix A in both branches; Fix B in if-branch only (matches CONTEXT.md).

2. **Should Fix A apply to `layout_selected` else-branch when no consumer is found?**
   - What we know: When `downstream_consumer is None`, `spine_x = root.xpos()` — the
     chain is placed at its current position. There is no consumer to measure clearance from.
   - Recommendation: Skip Fix A when consumer is None. Guard with `if consumer is not None:`.
