<decisions_made>

- Two-phase architecture: Phase 1 runs horizontal layout, Phase 2 runs vertical layout
  on the consumer with horizontal block nodes excluded via node_filter. This is the
  agreed fix — avoids modifying compute_dims/place_subtree signatures.
- Use fresh `memo_phase2 = {}` for Phase 2 to avoid stale dims from Phase 1 cache.
- `final_subtree_nodes` must root from `original_selected_root` (not rebound `root`)
  so bbox and state write-back cover all touched nodes.
- Bug 3 root cause: layout_upstream BFS rebinds root from m1 to m2, then ONLY runs
  horizontal layout, never calling place_subtree on m1. Phase 2 fixes this.
- Bug 2 root cause (still unverified in real Nuke): replay scan in
  `_place_output_dot_for_horizontal_root` correctly finds consumer via existing_dot —
  this was fixed in 11.1 and should be correct. Need end-to-end test to verify.
- Bug 1 root cause: `leftward_extent` formula uses `spine_nodes_ordered[i].screenWidth()`
  but does not account for side-subtree widths (left_extents). Documented as known
  limitation in code comments.
- The 11.1 `spine_y = consumer.ypos() - dot_gap - root.screenHeight()` formula is
  geometrically correct for placing m2 above m1 with a vertical wire to the dot.
</decisions_made>
