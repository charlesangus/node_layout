<current_state>
Two root-cause bugs identified and fixed this session (both in node_layout.py, 335/335 CI tests pass,
node_layout.py NOT YET COMMITTED):

1. **Horizontal layout still ignoring freeze** ŌĆö root cause: `_expand_scope_for_freeze_groups` only
   scanned selected (spine) nodes. Frozen blocks living entirely in side-input subtrees of spine nodes
   are never selected by the user, so their freeze UUIDs were never found ŌåÆ freeze_group_map was empty
   ŌåÆ `place_subtree_horizontal` called `place_subtree` on them with no freeze awareness.

   Fix: preliminary spine walk in `_layout_selected_horizontal_impl` collects side-input subtrees
   (and the primary input of the last spine node) into `_detection_scope`. This wider scope is passed
   to `_expand_scope_for_freeze_groups` + `_detect_freeze_groups`. Those groups are then handled by
   the existing offset restoration code.

2. **Vertical layout doesn't account for freeze block size** ŌĆö root cause: non-root freeze members
   are excluded from `node_filter`. So `compute_dims(block_root, node_filter)` treats the root as a
   leaf (no inputs), returning only `root.screenHeight()`. `place_subtree` allocates a band of that
   height for the root. After offset restoration places non-root members ABOVE the root, they collide
   with nodes in the bands above.

   Fix: `freeze_block_sizes` dict added as optional parameter to `compute_dims` and `place_subtree`.
   For freeze block roots with no in-filter inputs, returns `(root.screenWidth(), block_bounding_box_height)`
   instead of just root dims. `layout_selected` computes and passes this dict.
</current_state>
