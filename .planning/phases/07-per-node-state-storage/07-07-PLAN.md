<objective>
Wire the stored `h_scale` and `v_scale` per-node state into re-layout so that Layout Upstream
and Layout Selected apply the previously accumulated scale when re-running over nodes that were
shrunk or expanded.

Purpose: UAT test 4 reported "subsequent layouts do not respect the setting." Shrink/Expand
correctly accumulates scale in per-node state knobs, but the layout functions read only the
stored scheme — h_scale and v_scale are never consumed. This plan closes that gap.

Output: `compute_dims()` and `place_subtree()` gain `h_scale` and `v_scale` keyword parameters.
Layout entry points read these from per-node state and pass the root node's values to the geometry
engine. Memo key is extended to include the scale values.
</objective>

read_node_state() returns dict with keys: scheme, mode, h_scale, v_scale (all present, defaults 1.0).

KEY CONSTRAINT from STATE.md decisions:
- "State read at entry points only; never inside compute_dims() or place_subtree() — mid-recursion reads break memoization"
- This means: build per_node_h_scale and per_node_v_scale dicts at the entry point, then extract a single root_h_scale / root_v_scale float to pass to compute_dims/place_subtree (same pattern as per_node_scheme → root_scheme_multiplier).

KEY DESIGN for memo key:
- Current key: (id(node), scheme_multiplier)
- New key: (id(node), scheme_multiplier, h_scale, v_scale)
- All four values must be included because compute_dims results differ when h_scale/v_scale differ.

KEY DESIGN for applying scales in compute_dims and place_subtree:
- h_scale multiplies the horizontal margin: `int(_horizontal_margin(node, slot) * h_scale)`
- v_scale multiplies the vertical margin: `int(_subtree_margin(...) * v_scale)`
- v_scale also multiplies vertical_gap_between result: `int(vertical_gap_between(...) * v_scale)`
  EXCEPT when vertical_gap_between returns snap_threshold-1 (same-color tight gap) — do NOT
  scale that case, it represents a minimum snap constraint not a layout spacing preference.
- The snap_threshold-1 check is internal to vertical_gap_between, so: call vertical_gap_between
  normally, then apply v_scale only if the result > snap_threshold - 1. Alternatively, pass v_scale
  into vertical_gap_between as a new parameter and apply it inside, guarded by the same-color check.
  SIMPLER APPROACH: multiply at the call site in compute_dims/place_subtree:
    `gap = vertical_gap_between(inp, node, snap_threshold, scheme_multiplier)`
    `gap = max(snap_threshold - 1, int(gap * v_scale))` if not same tile color, else leave as-is.
  Because vertical_gap_between already returns snap_threshold-1 for same-color pairs, applying
  v_scale could shrink it below that minimum. Use `max(snap_threshold - 1, int(gap * v_scale))`.

RECURSIVE CALLS: compute_dims and place_subtree are recursive. The h_scale and v_scale passed
in come from the entry point's root node state. Use the same h_scale/v_scale for the entire
subtree rooted at that root (same as how scheme_multiplier works — it is the root's value, not
per-recursive-node). Per-node h_scale/v_scale variation across the subtree is a future feature;
for this plan, apply root-level h_scale/v_scale uniformly to all recursive calls.

All recursive internal calls to compute_dims and place_subtree within those functions must
also pass h_scale and v_scale through.
